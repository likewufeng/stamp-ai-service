## **镜像是不可变的，代码在构建时被“烧录”进去了**

### Dockerfile 里的关键一行

```
COPY . .
```

这行命令在 `docker compose build` 时执行，**把当前目录所有代码复制进镜像层**。构建完成后，镜像里的代码就**固定死了**。

------

### 运行时发生什么？

```
┌─────────────────────────────────────────────────────────────┐
│                    构建阶段 (build time)                     │
│  源码 ──COPY . .──▶ 镜像层 (只读，永久保存)                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    运行阶段 (run time)                       │
│  容器启动 ──▶ 从镜像读取代码 ──▶ python app.py               │
│  ❌ 宿主机改代码，容器里看不到（除非挂载卷）                 │
└─────────────────────────────────────────────────────────────┘
```

**原来的镜像里装的是旧代码**，启动新容器依然跑旧逻辑。

------

## 🔄 什么时候必须重建镜像？

| 变动类型            | 是否需重建 | 原因                         |
| ------------------- | ---------- | ---------------------------- |
| Python 代码 (`.py`) | ✅ **是**   | `COPY . .` 已烧录进镜像      |
| `requirements.txt`  | ✅ **是**   | `pip install` 在构建时跑     |
| `Dockerfile`        | ✅ **是**   | 构建逻辑变了                 |
| 系统依赖 (apt 包)   | ✅ **是**   | `apt-get install` 在构建时跑 |
| 模型文件下载        | ✅ **是**   | `RUN curl ...` 在构建时跑    |

------

## 💡 什么时候**不需要**重建？（开发模式）

用 **卷挂载** 覆盖容器内代码，实现热重载：

```bash
# docker-compose.override.yml (仅开发用)
services:
  stamp-ai:
    volumes:
      - .:/app          # 挂载宿主机代码覆盖镜像里的代码
      - ./data:/app/data
    command: python -m uvicorn app:app --host 0.0.0.0 --port 8000 --reload  # 开启 reload
    environment:
      - APP_RELOAD=true
```

效果：

- 宿主机改代码 → 容器内立即生效（uvicorn `--reload` 自动重启 worker）
- **不需要 `docker compose build`**

> ⚠️ **仅限开发环境**。生产环境**必须用镜像烧录代码**，保证：
>
> - 版本可追溯（镜像 = 发布物）
> - 多副本一致性
> - 回滚只需切镜像标签

## 🎯 类比理解

| 概念             | 类比                                   |
| ---------------- | -------------------------------------- |
| **Dockerfile**   | 食谱                                   |
| **Build**        | 做菜（把食材做成成品菜）               |
| **Image**        | **成品菜（装盘后不可改）**             |
| **Container**    | 食客吃这道菜的一个份                   |
| **源码改动**     | 换了食谱                               |
| **不重建直接跑** | 指望上一道菜自动变成新菜 —— **不可能** |

## 📋 生产环境标准流程（再强调一次）

```bash
# 1. 代码改完、测通过
git commit -am "fix: xxx"
git push

# 2. 服务器拉取 + 重建镜像（烧录新代码）
git pull
docker compose build stamp-ai    # 生成新镜像 stamp-ai-service_stamp-ai:latest

# 3. 滚动更新（旧容器优雅退出，新容器启动）
docker compose up -d --no-deps stamp-ai
```

**镜像就是交付物**，代码变了 → 必须生产新交付物。这正是 Docker “构建一次，到处运行” 的核心价值。
